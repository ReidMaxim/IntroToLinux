    22	
    23	--------------------------------------
    24	Console
    25	--------------------------------------
    26	In Linux the console is the actual system interface. It is the place where input/output happens. It differs from the terminal
    27	in that a terminal runs as a user application.
    28	It allows for direct communication with the kernel.
    29	(wiki.debian.org/Console)
    30	-----------------------------------------
    31	Terminal
    32	------------------------------------------
    33	A terminal is a program that pretends to be a console.
    34	Some include:
    35	GNOME Terminal
    36	Konsole
    37	xterm
    38	Alacritty
    39	
    40	A terminal gives the user access to a text terminal and all its applications such as:
    41	CLI (command line interface)
    42	TUI (text user interface)
    43	It is just a window you type into.
    44	It does not understand the command by itself.
    45	---------------------------------------------------------
    46	Shell
    47	---------------------------------------------------------
    48	The shell is the program that understands the text commands.
    49	Reads what you type
    50	Interprets it
    51	Runs it
    52	Handles pipes, variables, scripts, redirection, etc.
    53	
    54	--Flow of How It Works--
    55	Keyboard
    56	Terminal (window)
    57	Shell (bash, zsh, etc.)
    58	Linux kernel
    59	
    60	--Bash (Bourne Again SHell)--
    61	Default shell on most Linux systems
    62	Scriptable
    63	Very well documented
    64	Backward compatible with original Unix sh
    65	Very stable and predictable
    66	
    67	--Other shells:--
    68	zsh – better autocomplete and themes
    69	fish – user friendly, great defaults
    70	dash – very fast, minimal, used for system scripts
    71	ksh – older Unix shell
    72	
    73	--TTY--
    74	Ctrl + Alt + F1 opens a TTY (Teletypewriter).
    75	This opens a shell directly to the console.
    76	This is a console, not a window.
    77	
    78	--Command Line--
    79	The command line is a text-based interface.
    80	This refers to a style of interaction, not a program.
    81	
    82	--Terminal = tool
    83	Shell = interpreter
    84	Command line = method of interaction
    85	
    86	--What a Linux Command Is
    87	A Linux command is usually one of four things:
    88	Executable program
    89	Shell builtin
    90	Script
    91	Alias
    92	
    93	--When You Type a Command
    94	The shell has an order it follows:
    95	Checks if it is a built-in command (things like cd, exit, history)
    96	Checks to see if it is an alias
    97	Aliases are user created shortcuts (example: alias ll="ls -la")
    98	Checks if it is a function
    99	Searches the directories in $PATH for an executable by the name entered
   100	You could type out the full path of the executable, but $PATH is a shortcut.
   101	
   102	--Command Structure
   103	Command = what to do
   104	Flag = how to do it
   105	Argument = what to do it to
   106	
   107	An argument points the command at what it’s doing.
   108	A flag can be used to change how the command behaves.
   109	Flags usually start with - or --.
   110	If you have a file name that starts with -, you can use -- by itself to tell the command that everything past -- is an
   111	argument and not a flag.
   112	
   113	--Man Pages
   114	A man page is an in-depth breakdown of a command and how to use it.
   115	Example:
   116	man ls
   117	It explains:
   118	What the command does, How to use it, What flags exist, What arguments it expects
   119	No internet required. All local.
   120	Useful keys:
   121	q = quit
   122	/text = search
   123	n = next match
   124	
   125	
   126	--clear
   127	Clears visible content on the terminal.
   128	Keeps your scrollback intact.
   129	Command history untouched and does not reset terminal state.
   130	
   131	--reset
   132	Used to reinitialize the terminal to its default state.
   133	Used when programs crash or break line wrapping.
   134	Resets screen and keyboard connection
   135	Does not clear .bash_history
   136	Does not delete command history
   137	Does not log you out or kill the shell
   138	
   139	--exit
   140	In terminal window → closes window
   141	In TTY → logs you out
   142	In a script → stops execution
   143	
   144	--echo
   145	Creates an output.
   146	Example:
   147	echo "hey linux"
   148	Tells Linux to say it out loud.
   149	Used in scripts, pipelines, and debugging.
   150	
   151	--who
   152	Shows who is logged in right now.
   153	Use -a to display all info or -q for quick list.
   154	
   155	w
   156	Shows who is logged in and what they are doing.
   157	Shows system load and idle time.
   158	More detailed than who.
   159	Common flags:
   160	-h hide header
   161	-s short format
   162	
   163	whoami
   164	Will tell you user and if your privilege has been escalated.
   165	id
   166	What permissions do I have.
   167	Displays user ID, group ID, and all group members.
   168	Flags:
   169	-u UID only
   170	-g GID only
   171	-n show names instead of numbers
   172	Just permission truth.
   173	
   174	date
   175	Shows date and time.
   176	Can be formatted with flags.
   177	
   178	uname
   179	What kind of system am I on.
   180	Shows kernel and system info.
   181	Common flags:
   182	-a everything
   183	-r kernel release
   184	-m architecture
   185	Kernel info, not distro branding.
   186	
   187	pwd
   188	Where am I.
   189	Prints current active directory.
   190	-P displays physical path.
   191	ls
   192	List directory contents.
   193	Flags:
   194	-l long format
   195	-a show hidden files
   196	-h human readable sizes
   197	-R recursive
   198	Arguments: files, directories.
   199	
   200	shutdown
   201	Schedules a clean shutdown of the system.
   202	Flags:
   203	-h halt
   204	-r reboot
   205	now means now
   206	Arguments include time.
   207	shutdown now
   208	shutdown +5
   209	
   210	poweroff
   211	Shutdown and cuts power.
   212	reboot
   213	Restart system.
   214	halt
   215	Like shutdown but does not kill the power.

   222	## SHELL_METACHARACTERS
   223	
   224	SHELL METACHARACTERS
   225	============================================================
   226	
   227	Shell Metacharacters (MCs)
   228	These are characters the shell recognizes and treats differently than normal text.
   229	They can change how a command is interpreted before the command actually runs.
   230	The shell reads the metacharacter first and then uses it to control behavior.
   231	
   232	Examples:
   233	ls *.txt
   234	Lists all files ending in .txt in a directory.
   235	ls file[123].txt
   236	Lists file1.txt, file2.txt, file3.txt.
   237	ls file?.txt
   238	Matches exactly one character.
   239	
   240	Pipe |
   241	Sends output to another command.
   242	ls | grep txt
   243	
   244	Redirection
   245	> overwrite output
   246	>> append output
   247	ls > files.txt
   248	ls >> files.txt
   249	< input redirection
   250	Program reads input from file, not keyboard.
   251	sort < names.txt
   252	Command chaining
   253	
   254	; run a command after another always
   255	mkdir test; cd test
   256	
   257	&& run next command only if first succeeds
   258	make && sudo make install
   259	
   260	|| run next command only if first fails
   261	cd mydir || echo "Directory not found"
   262	
   263	& run in background
   264	sleep 60 &
   265	
   266	Variable expansion
   267	$ variable expansion
   268	echo $HOME
   269	
   270	Escape
   271	\ escape tells the shell the next character is not a metacharacter
   272	ls \*.txt
   273	
   274	Quotes
   275	' literal mode
   276	echo '$HOME' outputs $HOME
   277	" controls expansion
   278	echo "$HOME" expands but keeps it as one string
   279	Example:
   280	echo "Downloads"
   281	Returns Downloads as a string.

   288	## ROOT_SUPERUSER_SUDO
   289	
   290	
   291	ROOT / SUPERUSER / SUDO
   292	============================================================
   293	
   294	Root / Superuser / sudo
   295	Root
   296	User ID (UID) = 0
   297	Root has unlimited permissions.
   298	Can read any file, modify any system setting, kill any process.
   299	
   300	Superuser
   301	Any account with root-level privileges.
   302	Root is THE superuser.
   303	
   304	sudo
   305	Super User DO.
   306	Allows user to run a single command as root.
   307	Safer than logging in as root.
   308	Limits danger
   309	Creates audit logs
   310	Requires password
   311	Can restrict what commands users may run
   312	
   313	What happens when you use sudo
   314	You type your password.
   315	sudo checks /etc/sudoers.
   316	If allowed, command runs as root.
   317	Timeout is 5–15 minutes.
   318	
   319	sudo flags
   320	-l list what you are allowed to do
   321	-v refresh sudo timeout
   322	-k invalidate sudo (log out)
   323	-u user run as different user
   324	-i root login shell
   325	-s root shell (same window, same environment)
   326	
   327	Example sudo commands
   328	sudo apt install ani-cli
   329	sudo dnf update
   330	sudo nano /etc/fstab
   331	sudo cat /etc/shadow
   332	sudo systemctl restart ssh
   333	sudo systemctl enable apache2
   334	sudo useradd testuser
   335	sudo mount /dev/sdb1 /mnt
   336	sudo reboot
   337	sudo poweroff
   338	If it is your file, you do not need sudo.

   383	## HEADLESS_SERVER
   384	
   385	HEADLESS SERVER
   386	============================================================
   387	
   388	Headless Server
   389	A computer that runs without a monitor, keyboard, mouse, and usually no graphical desktop.
   390	Examples:
   391	Raspberry Pi
   392	A cloud VM
   393	A rack server in a datacenter
   394	
   395	Normally using SSH users can interact with the system.
   396	ssh username@hostname_or_ip
   397	
   398	It is important to know terminal commands because it is often the only way to interact with a headless server.
   399	
   400	examples:
   401	File management: ls, cd, cp, mv, rm
   402	Editing files: nano
   403	Managing services: systemctl start, enable, status
   404	Monitoring system: top, htop, df -h, free -m, journalctl

   412	## APPIMAGE_AND_FLATPAK
   413	
   414	APPIMAGE AND FLATPAK
   415	============================================================
   416	
   417	AppImage and Flatpak
   418	These are universal containerized packaging formats for Linux that bundle a package with its dependencies.
   419	
   420	AppImage
   421	A Linux app in a single file (like a Windows .exe).
   422	Download one file, make it executable, run it.
   423	No install, no package manager.
   424	Runs in user space.
   425	Does not install systemwide.
   426	chmod +x myapp.AppImage
   427	./myapp.AppImage
   428	
   429	Flatpak
   430	A universal packaging system.
   431	A sandbox app platform that works the same on every distro.
   432	Runs in a sandbox.
   433	Uses shared runtimes.
   434	Apps share a base (shared runtime) of bundled libraries and system components that multiple Flatpak apps can use.
   435	Apps are isolated from the system.
   436	Permissions are tightly controlled.
   437	flatpak install flathub org.gimp.GIMP
   438	flatpak run org.gimp.GIMP
   439	
   440	
   441	Comparison summary
   442	subject/topic: Preferred method
   443	Root required: Native
   444	Cross-distro: AppImage, Flatpak
   445	Sandboxed: Flatpak
   446	Auto updated: Native, Flatpak
   447	Disk usage: Native low, AppImage medium, Flatpak high
   448	Easy removal: AppImage, Flatpak
   449	Use native if:
   450	It is in your repos
   451	You care about performance and tight system integration
   452	Use AppImage if:
   453	You are quickly trying something new
   454	You do not want system changes
   455	You are running a headless or locked-down system
   456	Use Flatpak if:
   457	You want newer app versions
   458	You want security isolation
   459	You are on a distro with old repos

   674	EXPANSION OF IDEAS
   675	PROCESSES VS PROGRAMS
   676	============================================================
   677	
   678	A program is a passive object. It is a file on disk that contains instructions.
   679	
   680	A process is a program that is running. It exists in memory and is managed by
   681	the kernel.
   682	
   683	Each process has:
   684	- A PID (Process ID)
   685	- A parent process
   686	- A private memory space
   687	- An execution state
   688	
   689	Linux separates these so one program can be used multiple times at once.
   690	
   691	This allows the user or system to kill a process without affecting the program
   692	file and frees up memory. This enables:
   693	- Multi-user systems
   694	- Services restarting automatically
   695	- System stability
   696	
   697	Analogy:
   698	Program = blueprint
   699	Process = a building made from that blueprint
   700	
   701	
   702	============================================================
   703	FORK AND EXEC (HOW LINUX RUNS ANYTHING)
   704	============================================================
   705	
   706	Linux does not "launch programs". It uses a two-step process.
   707	
   708	1. fork()
   709	   - Creates a copy of the current process
   710	   - Parent and child run independently
   711	   - The child initially looks identical to the parent
   712	
   713	fork exists to provide:
   714	- Predictable behavior
   715	- Controlled duplication
   716	- Safe isolation
   717	
   718	2. exec()
   719	   - Replaces the current process with a new program
   720	   - PID stays the same
   721	   - Memory, code, and instructions are swapped out
   722	
   723	fork creates a process.
   724	exec turns it into something useful.
   725	
   726	Why Linux splits this:
   727	- Shells can launch commands and stay alive
   728	- Background jobs work correctly
   729	- One crash does not spread upward
   730	
   731	PID 1 forks and execs everything else.
   732	
   733	
   734	============================================================
   735	PROCESS STATES
   736	============================================================
   737	
   738	Most processes are not running.
   739	
   740	Common states:
   741	
   742	Running:
   743	- Actively using the CPU
   744	- Rare and brief
   745	
   746	Sleeping:
   747	- Waiting for I/O (disk, network, user input)
   748	- Most processes live in this state
   749	
   750	Waiting:
   751	- Waiting for another process or kernel event
   752	
   753	Stopped:
   754	- Paused (example: Ctrl + Z)
   755	
   756	Zombie:
   757	- Finished execution
   758	- Still has an entry in the process table
   759	- Waiting for the parent to acknowledge exit
   760	
   761	
   762	============================================================
   763	WHY ZOMBIES EXIST
   764	============================================================
   765	
   766	A child process remains a zombie until the kernel reports an exit code and
   767	completion status.
   768	
   769	The parent process "reaps" the child.
   770	
   771	Zombies are bookkeeping, not active processes.
   772	
   773	
   774	============================================================
   775	WHY PROCESS STATES MATTER
   776	============================================================
   777	
   778	These different states allow thousands of processes to exist where most are
   779	sleeping and cost the CPU almost nothing.
   780	
   781	This differs from Windows, where background processes are often heavier and more
   782	CPU-taxing.
   783	
   784	Linux also exposes zombie states, allowing you to see what happened after a
   785	process stops instead of hiding that information.
   786	
   787	These features excel in:
   788	- Server environments
   789	- Cloud setups
   790	- Systems where resources are used in bulk
   791	- Long-lived idle processes
   792	
   793	
   794	Process states are an optimized method the kernel uses to decide who or what
   795	needs the CPU right now.
