## HOW_LINUX_HANDLES_FILES

============================================================

Linux doesnt require file extension, on the OS/Kernel level mostly treats files as bytes,
Byte = 8 bits or an Octet from networking class.
bits are binary (1 and 0's) but bytes are represented as hexadecimal values. 4 bits make exactly make 1 hexadecimal value. 4 bits = a nibble

--At the lowest level (Kernel/OS interactions)--
Things that Matter. Permissions (can the user read/write/execute?)
File Meta data (owner, timestamps, etc) File contents (what the Bytes look like)
Whether or not the File is Executable (has execute bit, or a shebang line) Shebang line is a sequence of characters at the beginning of a script that specifies which interpreter should be used to execute the commands within the script.
(#! followed by absolute path to the interpreter)

Normally when you click on a .png file
it opens intended App. this process is normally handled by the desktop environment. it uses an "app-association system"
The fast and inaccurate way is to just check the file extension. easy and simple for humans or a file system
but anyone can rename a file to not have lets say a .png extension instead linux tools often inspect files for bytes at the beginning of Script.
These are called Magic bytes.

--MIME types--
MIME types are standardized labels
that web browsers or desktop environments use determine a files type. (image/png, text/plain, application/pdf Audio/mpeg
you double click a file in a file manager the desktop will figure out the mime type (often via shared libraries)
looks up preferred default app, launches app with file path.
this doesn't normally apply to launching from terminal
you could cat a .png and it will output some jumbled text, mime type is like a lable on a box to give an idea of contents

--Summary--
Magic bytes = Foundation. At beginning of file to id. format
Shebangline = the instruction- just for text file tells Kernel how to Execute or what interpreter to use to read and execute script.

--Comparison Table--
                 magic bytes              she bang              mime type
visible?         no (binary)              yes (first line)      no (metadata)
used by?         linux kernel/file cmd    linux kernel/shell    Browser/Desktop UI
example          0x7F 45 4C 46           #!/bin/bash           application/pdf

## FILE_EXTENSIONS

============================================================

What is A file extension? its a suffix after the last dot in a file name. in linux a file extension is just part of the name the Kernel does not rely on it to decide what a files type is VS windows that does rely on the files extension to know file type and associated programs. in linux its mostly a hint for humans and desktop environments.

Linux doesnt have an exact .exe equivalent.
instead decides something is runnable based on: Execute permission, File format (Binary VS Script), interpreter or loader availability.
examples would be: Compiled binaries. (a file with machine code that the computers cpu can execute directly.
Examples would be: ls, bash, grep, python, firefox... if it is an ELF binary and the execute bit is set... it will run
an execute bit (x) is a permission flag within linux filesystem's security controlling what actions a user can do to a file or directory "chmod +x filename" to add an execute bit.
ELF = Executable and Linkable format. its a standard file format for executable programs and libraries.

A file extension is a label on a box. Linux doesn't just trust the label it checks whats inside and whether your allowed to open and run it.

## EXECUTABLE_FILE_TYPES

============================================================

--.bin files--
.bin means binary but it is pretty vague. a .bin file might be a compiled executable, firmware, raw data, an installer payload. A .bin doesnt guarantee an executable. A .bin only runs if its an ELF binary or a script with a shebang line and it has an execute permission.

--.sh files (shell scripts)--
they are plain text contain shell commands and are executed by a shell (bash, sh, zsh, etc...) they rely on a shebang line (#!/bin/bash) the .sh is not important to have in file name needs execute permission.

--.py files (Python scripts)--
same idea as .sh but with a python interpreter.

--Comparison Table--
type       compiled        human readable    need interpreter    Typical use
.bin       normally yes    no                no                  Program/installers
.sh        no              yes               yes (shell)         Automation/sys tasks
.py        no              yes               yes (Python)        Apps/scripts/tools

bin = Machine code .sh, .py = plain text for another program to read.

## WILDCARD_METACHARACTERS

============================================================

Wildcard MC's are used for filename expansion (globbing) in the shell.
They match file names in the current directory or specified path.
the Shell expands them before the command executes

--How this differs from normal meta characters--
Regular MC: controls shell behavior handles redirection, piping, command chaining affect how commands interact.
Wildcard Metacharacters: Only match file names, used in globbing expand into actual file names.

--Common WCMC--
* = (matches zero or more characters)
? = (matches only one character)
[ ] = (matches a range or set of characters)
[! ] = Negation (matches anything except)

## FILE_NAMING_BEST_PRACTICES

============================================================

How MCs Restrict naming file

MCs have special meaning to the shell. If used in a file name & the shell may interpret them instead of treating them literally, you can use a backslash "\" to tell the shell that the next character should be treated literally.
i.e. "rm budget202520252025.txt". Some commonly used naming practices: Lowercase letters when possible (stay consistent) Numbers, Hyphens and underscores. Avoid using "-" as the first or "leading" dash the shell will interpret your text as a flag. Avoid spaces and shell MCs altogether if possible (* ? [ ] { } ( ) & | ; > < $ " ' ~ )

Pick your Pattern and stick with it ReidMaxim, Reid_Maxim, reid_maxim. Use ISO date format YYYY-MM-DD. Avoid strange text/characters/emojis stick to plain ascii. Plan for simple and easy usage later on, considering globbing will make using wildcards in the future much easier i.e "report_2025_07.txt" "rm report_2025_*.txt"

## LINKED_FILES

============================================================

What is a linked file?

A linked file is another name that refers to an existing files data. It allows multiple file names to point to the same content on disk. There are two types,

--Hard link--
Another filename pointing to the same "inode" (inode: index node = A file ID card… who owns it, permissions, file size, timestamps, where data is located on disk) a hardlink shares the exact same data on disk, indistinguishable from the original file. How to create: "ln original.txt copy.txt"

to view these in terminal run "ls -l" and expect to see:

"-rw-r--r-- 2 user user 1234 Jan 6 10:00 original.txt"
"copy.txt"

[type+permissions] [link count] [owner] [group] [size] [date] [name]

Link count shows how many files in directory point to the same inode

--Symbolic Link--
these are shortcuts, it points to the other files path, can point to directories, breaks if original file is deleted. to create one "ln -s original.txt shortcut.txt"

to view in terminal run "ls -l":

"lrwxrwxrwx 1 user user 12 jan 6 10:05 shortcut.txt -> original.txt"

## PIPED_FILES_FIFO

============================================================

A named pipe is a special file used for communication between programs. it lets one program send data directly to another. Think of it as a one way tunnel, one program writes and another reads. it doesnt store data permanently.

Created with: "mkfifo mypipe". Some example usage would be streaming logs between programs, connect to background service that listens to the pipe. it also allows programs to interact using a pipe but they dont have to start at the same time.

## TEXT_FILES_IN_LINUX

============================================================

--How Common are simple text files--
both are used in windows and linux. plays way more of a role in linux. system configs, logs, scripts, service definitions are all text files. falls back onto the core linux philosophy that everything is a file. windows uses some of these same files (logs, configs, scripts) but many settings are not in a text file but instead are stored in a registry, config files are often hidden from users

--.conf vs .txt vs .csv--
.txt: Plain text, No structure enforced… often used for notes, logs, documentation, quick export.
.conf: Configuration files… Still plain text but structured for software to read. .conf implies: the file changes system behavior… caution when editing.
.csv: (Comma Separated Values) Structured text that represents tabular data. Used for spreadsheets, database exports, data analysis and bulk imports. commonly used in data or business.

--Other Plain text file types--
.log = log file
.yaml/.yml = config docs
.json = structured data
.xml = structured markup
.md = markdown
.sh = shellscript
.py = python

--Exporting databases to .txt or .CSV--
both can be done and theres instances where one would be better then the other but most of the time a .CSV file would be the better option.

Strengths: human readable, portable, universally supported, easy to grep

Weakness: no data typing (everything is text) no indexing, slow for big datasets

Dangers: data corruption (commas inside field not properly quoted = broken rows) excel command injection. excel could interpret data as a formula or command (malicious CSV's are a thing). No encryption exporting full database could expose sensitive data.

Performance: large datasets can make millions of rows with no indexing or query optimization could make parsing that much data tough.

## GREP

============================================================

Grep (global regular Expression print)

Searches for patterns in text and prints matching lines.

"grep "error" file.txt" — prints any line containing the word error.

--Basic grep flags--
-i = ignore case
-n = show line numbers
-v = invert (show lines that dont match)
-r = recursive (search folders)
-c = count matches
-w = match whole word only
-A 3 = show 3 lines after match
-B 3 = show 3 lines before match
-C 3 = show before and after 3 lines

--Examples--
grep "error" app.log — search logs for "error" make it case insensitive
grep -i "error" app.log matches error, ERROR, ErRoR.
grep -n "error" app.log show error and the number line its on.
grep -c "error" app.log counts errors and prints a number
grep -C 2 "error" app.log show 2 lines before and after error.

## COMMAND_PIPING

============================================================

Command Piping

take output of left command and feed it to right command

i.e: cat file.txt | grep "error" this prints file, send it to grep, only show lines with error. Not a super useful command because you could just grep the file directly

live errorlog: tail -f app.log | grep "error"

head, tail, less and more are used often as output limiters or viewers, could be thought of as filters on the end of a pipe chain.

## TAB_COMPLETION

============================================================

When you press tab the shell checks what directory am I in? what files and folders exist here and what ones start with what has currently typed in. this is the process for completing files and directories.

Shell uses similar steps for finding commands it will reference all directories in $PATH and find commands starting with what you typed.

Tab completion increases speed and accuracy and reduces cognitive load

Speed: dont have to type full names
Accuracy: avoids typos
Discovery: press tab twice to see list of matching options
